KEY ISSUES IDENTIFIED
=====================

1. INCONSISTENT ARCHITECTURE PATTERNS
---------------------------------------
Issue: collection.controller.js exports functions instead of a class instance
Issue: announcements.controller.js uses static methods instead of instance methods
Impact: Inconsistent usage patterns across the codebase

2. DUPLICATE FUNCTIONALITY
---------------------------
Issue: Both department.controller.js and departments.controller.js exist with overlapping functionality
Impact: Confusion about which controller to use, potential maintenance issues

3. SECURITY CONCERNS
--------------------
Issue: telegram.controller.js exposes verification codes in API responses (lines 352, 401)
Issue: telegram.controller.js has hardcoded phone numbers (lines 319, 371)
Issue: Some controllers build dynamic SQL queries without proper sanitization
Impact: Security vulnerabilities, data exposure

4. INCONSISTENT RESPONSE FORMATS
---------------------------------
Issue: Some controllers return { success: true, data: ... }, others return { data: ... }
Issue: announcements.controller.js returns { error: ... } instead of { success: false, error: ... }
Impact: Inconsistent frontend error handling

5. MISSING INPUT VALIDATION
---------------------------
Issue: Only announcements.controller.js uses express-validator
Issue: Most controllers lack proper input validation
Impact: Potential invalid data entry, security risks

6. INCONSISTENT PERMISSION CHECKING
-----------------------------------
Issue: Permission checks are scattered and inconsistent
Issue: Some controllers check roles inline, others don't check at all
Impact: Potential unauthorized access

7. CODE DUPLICATION
------------------
Issue: Similar pagination logic repeated across controllers
Issue: Similar error handling patterns repeated
Impact: Maintenance burden, inconsistent behavior

8. NAMING INCONSISTENCIES
--------------------------
Issue: Mix of camelCase and snake_case in database queries
Issue: Inconsistent parameter naming (e.g., departmentId vs department_id)
Impact: Confusion, potential bugs


REFACTORING RECOMMENDATIONS
============================

PRIORITY 1: CRITICAL SECURITY FIXES
------------------------------------
1. Remove verification codes from API responses in telegram.controller.js
2. Remove hardcoded phone numbers from telegram.controller.js
3. Add input validation middleware to all controllers
4. Sanitize all dynamic SQL queries

PRIORITY 2: ARCHITECTURE CONSISTENCY
-------------------------------------
1. Convert collection.controller.js to class-based pattern
2. Convert announcements.controller.js static methods to instance methods
3. Decide on single department controller (merge or deprecate one)
4. Standardize response format across all controllers

PRIORITY 3: CODE QUALITY
--------------------------
1. Create shared middleware for common operations (pagination, validation)
2. Create shared permission checking utilities
3. Extract common error handling into helper functions
4. Standardize naming conventions

PRIORITY 4: MAINTAINABILITY
-----------------------------
1. Add JSDoc comments to all controller methods
2. Create base controller class with common functionality
3. Implement consistent logging across all controllers
4. Add unit tests for critical controllers


DETAILED FINDINGS BY CONTROLLER
===============================

TELEGRAM.CONTROLLER.JS (570 lines)
-----------------------------------
Security Issues:
- Lines 352, 401: Verification codes exposed in API responses
- Lines 319, 371: Hardcoded phone numbers (+254736075771, +254724363290)
- Global Map used for verification codes (should use Redis with TTL)

Architecture Issues:
- Uses TelegramService for business logic (good separation)
- Cache health and refresh methods reference galleryCache helper

ANNOUNCEMENTS.CONTROLLER.JS (350 lines)
----------------------------------------
Architecture Issues:
- Uses static methods instead of instance methods (inconsistent with other controllers)
- Uses express-validator for input validation (good practice, but not used elsewhere)
- Response format uses { error: ... } instead of { success: false, error: ... }

COLLECTION.CONTROLLER.JS (575 lines)
------------------------------------
Architecture Issues:
- Exports individual functions instead of a class instance
- Inconsistent with all other controllers in the codebase
- Should be converted to class-based pattern

DEPARTMENT VS DEPARTMENTS CONTROLLERS
--------------------------------------
- department.controller.js (949 lines): More comprehensive, includes uploads, dashboard, communications
- departments.controller.js (371 lines): Simpler CRUD operations
- Recommendation: Merge into single controller or clearly separate responsibilities

RESPONSE FORMAT INCONSISTENCIES
--------------------------------
Standard format used by most controllers:
{ success: true, data: ... }
{ success: false, error: ... }

Variations found:
- announcements.controller.js: { error: ... }
- sms.controller.js: { success: true, providers: ... } (uses different key names)
- reports.controller.js: { success: true, reports: ... }

PERMISSION CHECKING INCONSISTENCIES
------------------------------------
Some controllers check permissions:
- announcements.controller.js: Checks Super Admin, Pastor, First Elder roles
- approvals.controller.js: No explicit permission checks
- department.controller.js: Role-based access control

Many controllers have no permission checks at all
